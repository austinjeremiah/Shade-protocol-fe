## MPC — Production Gaps (current implementation is testnet-grade)

| Item | Detail |
|---|---|
| **Persistent committee keypairs** | Keys are ephemeral (regenerated on restart). Batches from a previous session cannot be re-verified after restart. For production, keys must be stored in DB or sealed env so existing signed batches remain verifiable across restarts. |
| **Real Merkle root in `mpc_settle`** | Currently passes `new_root = 0x0000...0000`. The contract accepts it and emits the event, but the on-chain root is zeroed. The relayer must compute the real post-insert lean-IMT root (insert both output commitments) and pass it to `mpc_settle`. |
| **Output commitments are `recipient_commitment`** | Current DB stores `recipient_commitment` as the output. The proper derivation: for each matched intent, the counterparty's `note_commitment` becomes the output commitment (i.e. intent A's output = intent B's input note, and vice versa). Review and tighten this mapping. |
| **Real intent amounts via Shamir** | Integration test uses random nullifiers/commitments. A full e2e with real deposited notes means the nullifiers are the actual input note nullifiers, and the output commitments are freshly generated by the protocol (not random). |
| **Settler triggered from committee loop** | Currently the settler must be triggered manually or via the in-process loop. In production, the committee server must reliably be started with `DATABASE_URL` to activate the settler automatically. |
| **Multi-batch support** | Settler creates one job per match per batch. If a batch has N matches, N jobs are queued. The relayer handles each independently. Verify atomicity guarantees across multi-match batches. |

| Circuit | What it must prove |
|---|---|
| `rfq_settlement` (standalone) | Note ownership + nullifier + accepted quote hash + intent hash + solver ID + fill constraints + output commitment + fee + policy |
| `compliance_membership` | ASP allow-root membership + deny-root non-membership + policy ID + validity window |
| `proof_of_fill_claim` | Intent hash + fill receipt hash + solver ID + destination tx hash + amount + recipient + deadline + quote hash |
| `remit_settlement` | Note ownership + nullifier + quote ID hash + payout corridor + amount + policy |
| **`mpc_settlement`** | **Proves the MPC batch output is consistent with the input note commitments — binds nullifiers + output commitments + batch hash + committee threshold + policy** |
